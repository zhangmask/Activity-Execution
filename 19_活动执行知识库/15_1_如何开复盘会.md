# 如何开复盘会

> **适用场景**：黑客松活动结束后，组织团队进行系统性复盘，总结经验教训。
> **前置条件**：请先完成 `01_事务统筹与团队分工.md`，了解团队组织结构和各组职责。

---

## 第一步：确定复盘时间线

### 复盘时间节点安排

| 时间 | 复盘内容 | 参与人员 | 形式 |
|------|---------|---------|------|
| 活动结束当天 | 各组内部快速复盘 | 各组全员 | 组内小会（30分钟） |
| 活动结束后1-3天 | 收集数据和反馈 | 各组组长 | 线上收集 |
| 活动结束后3-5天 | 全团队复盘大会 | 全体工作人员 | 线下/线上会议（2-3小时） |
| 活动结束后1-2周 | 撰写复盘报告 | 核心团队 | 协作文档 |
| 活动结束后2-4周 | 知识沉淀和存档 | 核心团队 | 文档归档 |

### 关键原则

- **趁热打铁**：活动结束后3天内完成核心复盘，记忆最清晰
- **对事不对人**：复盘是为了改进流程，不是追究责任
- **数据说话**：用数据支撑结论，避免主观臆断
- **全员参与**：每个人的声音都很重要

## 第二步：设计复盘会议议程

### 全团队复盘大会议程（2-3小时）

| 时间段 | 环节 | 内容 | 负责人 |
|--------|------|------|--------|
| 0:00-0:10 | 开场 | 复盘目的说明、规则宣导 | 总负责人 |
| 0:10-0:20 | 数据回顾 | 关键数据展示（报名、到场、满意度等） | 数据组 |
| 0:20-0:50 | 各组汇报 | 每组10分钟，汇报亮点、问题、建议 | 各组组长 |
| 0:50-1:00 | 茶歇 | 休息和自由交流 | - |
| 1:00-1:40 | 专题讨论 | 针对核心问题分组讨论 | 全员 |
| 1:40-2:10 | 方案共创 | 针对问题提出改进方案 | 全员 |
| 2:10-2:30 | 总结行动 | 形成行动项和负责人 | 总负责人 |
| 2:30-2:40 | 致谢 | 感谢所有参与者的付出 | 总负责人 |

### 复盘会规则（开场时宣导）

```
本次复盘会遵循以下原则：
1. 对事不对人——我们讨论的是流程和方案，不是个人
2. 用数据说话——尽量提供具体数据和事实
3. 每个人都发言——鼓励所有人表达观点
4. 聚焦改进——不仅说问题，更要说解决方案
5. 保密原则——复盘内容仅限团队内部
```

## 第五步：数据总结方法

### 核心数据指标

| 维度 | 指标 | 数据来源 |
|------|------|---------|
| 参与规模 | 选手报名数、实际参赛数、游客人数 | 报名系统、签到系统 |
| 传播效果 | 官网访问量、社交媒体曝光量、直播观看人数 | 统计后台 |
| 选手体验 | 选手满意度评分（1-10分） | 赛后问卷 |
| 嘉宾反馈 | 嘉宾满意度、嘉宾出席率 | 访谈/问卷 |

| 团队效率 | 任务完成率、延期任务数 | 项目管理工具 |
| 问题数量 | 现场突发问题数、解决率 | 现场记录 |

## 第六步：知识沉淀流程

### 知识沉淀步骤

```
第一步：收集
  - 各组复盘报告
  - 个人复盘报告
  - 活动数据汇总
  - 满意度调查结果
  - 现场照片和视频
  - 财务报表

第二步：整理
  - 按模块分类归档
  - 提取关键经验教训
  - 形成"做与不做"清单
  - 整理最佳实践案例

第三步：文档化
  - 撰写活动总结报告
  - 更新知识库文档
  - 制作新人培训材料
  - 整理供应商/合作伙伴评价

第四步：存档
  - 所有文档存入共享云盘
  - 按活动届次建立文件夹
  - 设置访问权限
  - 确保下一届团队可查阅

第五步：分享
  - 在团队内部分享会上传播
  - 在社群/公众号发布活动总结
  - 与合作伙伴分享成果
```

## Checklist：复盘准备清单

- [ ] 所有文档存入共享云盘

## 常见问题

**Q1：大家不愿意在复盘会上说真话怎么办？**
A：①采用匿名问卷先收集问题；②复盘会开场强调"对事不对人"原则；③领导先带头自我批评；④使用"便利贴法"——每人匿名写3个问题贴在白板上。

**Q2：复盘会变成诉苦大会怎么处理？**
A：主持人及时引导，将话题从"抱怨"转向"改进"。可以使用"问题→原因→方案"三步法，每提出一个问题必须附带一个改进方案。

**Q3：数据不全怎么办？**
A：①尽量收集现有数据；②对于缺失数据，记录缺失原因并在下次改进数据收集流程；③用定性反馈（访谈、问卷开放题）弥补定量数据不足。

**Q4：复盘报告写完没人看怎么办？**
A：①复盘报告要简洁，控制在10页以内；②突出"关键发现"和"行动项"；③在下届筹备启动时作为第一份阅读材料；④将核心结论编入新人培训材料。

### 真实案例：历史活动复盘教训

**6条核心教训**：
1. **经费控制**：最后会超很多很多，必须严格控制
2. **重心在线上**：Hackathon不能只是线下办活动，重点是线上自媒体撬动社区杠杆
3. **大型活动分组方式**：应按职能分组——线下执行团队、外联与合作团队、设计叙事团队、活动方案团队（线上）；按时间线分组工作，每个工作精确到人
4. **核心团队+机动团队**：核心团队好好维护，大机动团队赛前一个月启动，此时大量招募志愿者（比你想的数量还要多）
5. **按时间线梳理分工文档**：尽早梳理出按时间线排序、明确分工到人的文档，每个小模块一个主负责人，只盯着他
6. **统一文档管理**：文档不能乱七八糟，必须有一个超级强有力的管理人员，数一不二就用这个文档，不要再加别的

## 历史教训

- 历史活动复盘通常在活动结束后1周内完成，趁团队记忆最清晰
- 建议使用"4L法"复盘：Liked（喜欢的）、Learned（学到的）、Lacked（缺少的）、Longed for（期望的）
- 历史活动的数据复盘重点关注：选手完成率、项目质量、社区增长
- 复盘报告建议控制在A4纸10页以内，太长没人看
- 历史教训表明：个人复盘比团队复盘更能发现深层问题
- 知识沉淀是复盘最重要的产出，确保下一届团队能站在更高的起点上
