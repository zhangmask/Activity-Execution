# 如何设计报名表和审核选手

> **适用场景**：hackathon活动需要设计线上报名表收集选手信息，并建立审核流程筛选参赛者。
> **前置条件**：已完成线上平台搭建（参考 14_网站与线上平台.md）。

---

## 第一步：设计报名表

### 报名表设计要素

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| **基本信息** | | | |
| 姓名 | 文本 | 是 | 真实姓名 |
| 性别 | 单选 | 是 | 男/女 |
| 手机号 | 文本 | 是 | 联系方式 |
| 微信号 | 文本 | 是 | 拉群用 |
| 邮箱 | 文本 | 是 | 发送通知 |
| 身份证号 | 文本 | 否 | 购买保险用（如需要） |
| **教育/职业信息** | | | |
| 学校/单位 | 文本 | 是 | - |
| 专业/岗位 | 文本 | 是 | - |
| 年级 | 单选 | 否 | 大一/大二/大三/大四/研一/研二/已工作 |
| **参赛信息** | | | |
| 参赛身份 | 单选 | 是 | 个人参赛/团队参赛 |
| 团队名称 | 文本 | 条件必填 | 团队参赛时必填 |
| 团队成员 | 文本 | 条件必填 | 列出所有成员姓名 |
| 技术栈 | 多选 | 是 | 前端/后端/移动端/AI/设计/产品/硬件/其他 |
| 参赛经验 | 单选 | 否 | 首次参加/参加过1-2次/参加过3次以上 |
| 项目想法 | 长文本 | 否 | 简要描述你的项目想法（100字内） |
| **后勤信息** | | | |
| T恤尺码 | 单选 | 是 | S/M/L/XL/XXL |
| 饮食禁忌 | 多选 | 否 | 无/素食/清真/花生过敏/海鲜过敏/其他 |
| 是否需要住宿 | 单选 | 是 | 是/否 |
| 住宿偏好 | 单选 | 条件必填 | 可接受拼住/需要单住 |
| **其他** | | | |
| 如何得知活动 | 多选 | 否 | 朋友圈/公众号/学校社团/朋友推荐/其他 |
| 对活动的期待 | 长文本 | 否 | 开放性问题 |
| 特殊需求 | 长文本 | 否 | 无障碍需求等 |

### 飞书问卷 vs 腾讯问卷

| 对比维度 | 飞书问卷 | 腾讯问卷 |
|----------|----------|----------|
| 优点 | 与飞书生态打通、数据自动同步表格 | 功能强大、模板丰富、免费 |
| 缺点 | 需要飞书账号 | 与企业微信打通不如飞书 |
| 适用场景 | 团队使用飞书生态 | 通用场景 |
| 条件逻辑 | 支持 | 支持 |
| 数据导出 | 自动同步飞书表格 | 导出Excel/CSV |

## 第二步：设计审核流程

### 审核流程图

```
选手提交报名表
    ↓
自动确认邮件（收到报名）
    ↓
人工审核（1-3个工作日）
    ↓
├── 通过 → 发送通过通知 → 拉入选手群
├── 不通过 → 发送不通过通知（说明原因）
└── 待定 → 发送补充信息通知
    ↓
选手确认参加（回复确认）
    ↓
加入正式选手名单
```

### 审核标准

| 审核维度 | 评估要点 | 权重 |
|----------|----------|------|
| **信息完整性** | 所有必填字段是否填写完整 | 基础门槛 |
| **身份匹配** | 是否符合活动面向的群体（学生/职场人等） | 30% |
| **技术能力** | 技术栈是否匹配、是否有相关经验 | 25% |
| **参赛动机** | 对活动的期待、项目想法 | 20% |
| **团队构成** | 团队是否完整、成员技能是否互补 | 15% |
| **多样性** | 背景多样性、学校分布 | 10% |

### 审核决策矩阵

| 情况 | 决策 | 说明 |
|------|------|------|
| 信息完整 + 符合要求 | ✅ 通过 | 正常录取 |
| 信息完整 + 部分不符 | ⏸️ 待定 | 联系确认后决定 |
| 信息不完整 | ❓ 补充 | 联系补充信息 |
| 明显不符合要求 | ❌ 不通过 | 说明原因 |

## 第四步：管理报名数据

### 数据管理规范

| 管理事项 | 说明 |
|----------|------|
| **数据存储** | 使用飞书表格/腾讯文档统一管理，避免本地Excel散落 |
| **数据备份** | 每周备份一次数据到本地 |
| **数据安全** | 仅核心团队可访问完整数据（含身份证号等隐私字段） |
| **数据使用** | 仅用于活动相关用途，不外泄 |
| **数据删除** | 活动结束后3个月内删除敏感个人信息 |

## 常见问题

**Q：报名人数太多怎么办？**
A：①设置报名上限，先到先得；②增加审核标准，择优录取；③增设候补名单；②考虑增加场次或场地容量。

**Q：报名人数不足怎么办？**
A：①延长报名时间；②加大推广力度（校园海报、社交媒体）；③降低报名门槛；④联系合作院校推广。

**Q：有人报名后不参加怎么办？**
A：①要求"确认参加"才能保留名额；②设置确认截止时间；③准备候补名单；④活动前1天再次确认。

**Q：如何防止"僵尸选手"占名额？**
A：①报名后48小时内要求确认；②未确认自动释放名额给候补；③活动前3天再次确认出席。

### 真实案例：历史活动选手审核标准

**审核流程（2轮）**：
- 第一轮：目标控制人数在120人左右，定一个通过分数线，让100人在分数线之上
- 第二轮：剩余未录取选手按分数排序录取，最终录取230-250人（防止鸽子）

**评分标准（总分100分）**：

| 评分维度 | 分值 | 高分档(30-40分) | 中分档(15-29分) | 低分档(0-14分) |
|---|---|---|---|---|
| 过往成就与能力 | 40分 | 高质量产出：作品多、idea好、GitHub高star、设计作品集精美、曾上线产品、有知名企业任职、其他黑客松获奖 | 产出不多但有明确想法，知道下一步计划 | 概念模糊、堆概念词汇、不清楚在做什么、纯AI生成 |
| 当前项目执行力与思考 | 40分 | 目标明确、强执行力、能说明解决什么问题、idea好、推进速度快 | 有具体想法或早期探索阶段，知道下一步干啥 | 概念模糊、不清楚现阶段在干啥、纯AI生成 |
| 调性是否贴合 | 20分 | 表达极具生命力和热情、对build很热爱、热爱自己做的事和生活 | 中等，表达清晰 | 明显AI生成、全部套话、态度差、答非所问 |

## 历史教训

在hackathon活动中，报名和审核建议：
- **报名表精简**：hackathon选手不喜欢填太长的表单，必填字段控制在10个以内
- **技术栈字段**：非常重要，帮助选手了解团队技能分布，也方便后续组队匹配
- **快速审核**：hackathon报名周期通常较短，审核要在1-3天内完成
- **确认机制**：设置确认环节非常重要，能有效过滤"僵尸选手"
- **数据看板**：用飞书表格做一个数据看板，实时查看报名人数、审核状态、确认率等
