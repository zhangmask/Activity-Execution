# 如何管理项目提交

> **适用场景**：hackathon活动结束前，需要管理选手的项目提交，确保提交内容完整、格式统一。
> **前置条件**：已完成组队管理（参考 11_2_如何管理组队）。

---

## 第一步：确定提交内容要求

### 提交内容清单

| 提交项 | 格式要求 | 必填 | 说明 |
|--------|----------|------|------|
| **项目名称** | 文本 | 是 | 简洁有力的项目名称 |
| **项目简介** | 200字以内 | 是 | 一句话描述+详细说明 |
| **项目演示视频** | MP4，3分钟以内 | 是 | 演示项目功能和亮点 |
| **项目代码** | GitHub仓库链接 | 是 | 含README说明文档 |
| **项目PPT/演示文稿** | PDF/PPTX | 是 | 路演用，10页以内 |
| **项目截图** | PNG/JPG，3-5张 | 建议 | 展示项目界面和功能 |
| **技术文档** | Markdown/PDF | 否 | 技术架构和实现说明 |
| **其他材料** | 任意 | 否 | 设计稿、商业计划书等 |

### 提交平台选择

| 平台 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **飞书收集表** | 与飞书生态打通 | 需要飞书账号 | 团队使用飞书 |
| **腾讯文档** | 通用、免费 | 文件大小限制 | 通用场景 |
| **Devpost** | 国际化hackathon平台 | 英文界面 | 国际hackathon |
| **自建提交系统** | 完全自定义 | 开发成本高 | 大型活动 |
| **邮件提交** | 简单 | 管理困难 | 应急方案 |

## 第二步：制定提交时间线

### 提交时间线

| 时间节点 | 事项 | 说明 |
|----------|------|------|
| 活动开始时 | 公布提交要求 | 在开幕式上详细说明 |
| 活动中期 | 提交提醒 | 提醒选手注意时间，准备提交材料 |
| 提交截止前2小时 | 最后提醒 | 群内通知，强调截止时间 |
| 提交截止前30分钟 | 紧急提醒 | 最后一次提醒 |
| 提交截止时间 | 关闭提交通道 | 严格截止，不再接受新提交 |
| 提交截止后30分钟 | 提交确认 | 确认所有团队已提交，联系未提交团队 |

### 提交倒计时提醒模板

**提交前2小时提醒**：
```
⏰ 项目提交倒计时2小时！

亲爱的选手们，
距离项目提交截止还有2小时（截止时间：XX:XX）。

📋 提交checklist：
- [ ] 项目简介（200字以内）
- [ ] 演示视频（3分钟以内，MP4格式）
- [ ] GitHub仓库链接（含README）
- [ ] 演示PPT（10页以内）

📝 提交链接：[链接]

⚠️ 请务必在截止时间前完成提交，逾期不候！
```

**提交截止提醒**：
```
🚨 项目提交截止！

提交通道将在15分钟后关闭（XX:XX）。

请尚未提交的团队立即提交！
已提交的团队请确认提交内容完整。

提交链接：[链接]
```

## 第三步：制定提交检查清单

### 提交完整性检查清单

| 检查项 | 检查标准 | 状态 |
|--------|----------|------|
| 项目名称 | 非空、不超过20字 | ☐ |
| 项目简介 | 200字以内、包含项目说明和亮点 | ☐ |
| 演示视频 | MP4格式、3分钟以内、可正常播放 | ☐ |
| 代码仓库 | GitHub链接可访问、含README | ☐ |
| 演示PPT | 10页以内、可正常打开 | ☐ |
| 团队信息 | 团队名称、成员姓名完整 | ☐ |
| 联系方式 | 至少一个联系方式（手机号/微信） | ☐ |

### 提交质量检查清单

| 检查项 | 检查标准 | 状态 |
|--------|----------|------|
| 视频质量 | 画面清晰、声音清楚 | ☐ |
| 代码质量 | 有基本的代码规范、有注释 | ☐ |
| README | 包含项目说明、安装步骤、使用方法 | ☐ |
| PPT质量 | 排版清晰、内容完整、无明显错误 | ☐ |

## 第四步：设计评分表

### 评分维度设计

| 评分维度 | 权重 | 评分标准 | 分值范围 |
|----------|------|----------|----------|
| **创新性** | 25% | 项目创意是否新颖、是否有独特视角 | 1-10分 |
| **完成度** | 25% | 项目是否可运行、功能是否完整 | 1-10分 |
| **技术难度** | 20% | 技术实现是否有挑战性 | 1-10分 |
| **用户体验** | 15% | 界面设计、交互体验是否友好 | 1-10分 |
| **商业价值** | 10% | 项目是否有实际应用场景和商业潜力 | 1-10分 |
| **演示表现** | 5% | 路演表达是否清晰、有说服力 | 1-10分 |

### 评分流程

1. **评委培训**：活动前向评委说明评分标准和流程
2. **独立评分**：每位评委独立打分，不讨论
3. **分数汇总**：去掉最高分和最低分，取平均分
4. **排名确认**：按总分排名，确认获奖名单
5. **争议处理**：如有争议，由评审团讨论决定

## 常见问题

**Q：选手提交的视频太大怎么办？**
A：①限制视频大小（建议200MB以内）；②提供视频压缩工具推荐；③允许上传到B站/YouTube后提交链接。

**Q：有团队超时提交怎么办？**
A：①明确规定"超时不候"，维护规则的严肃性；②如有特殊情况（如网络故障），酌情给予5-10分钟缓冲；③超时提交的项目不参与评奖。

**Q：评委评分差异太大怎么办？**
A：①去掉最高分和最低分后取平均；②如果某项目评分差异特别大（如最高9分最低3分），组织评委讨论后重新评分。

**Q：代码仓库是私有的怎么办？**
A：①要求提交时将仓库设为公开；②如果涉及敏感信息，可以创建一个只含展示代码的公开仓库；③活动结束后可以设为私有。

### 真实案例：历史活动项目提交清单

**1. GitHub代码仓库（必做）**
- 上传代码至GitHub并公开仓库
- 设置仓库Tag为活动专属标签
- 商业项目请开源部分代码，并附带已部署实例
- Github Readme中需清楚撰写：项目技术栈、技术选型及框架
- 软件应用提供可直接访问的URL或安装包链接；硬件产品提供样品Demo

**2. 小红书产品发布视频（必做）**
- 发布平台：小红书
- 必带标签：活动专属标签×3
- 视频格式：时长1-3分钟，包含人像解说段落，至少展示2个Demo Case，关键讲解部分添加字幕
- 由队伍任意一位选手发布即可

**3. 种子用户访谈清单（选做）**
- 内容需包括：产品种子用户定义文档、个性化DM话术示例与发送记录、用户访谈记录与分析
- 以飞书文档链接格式提交

**4. 项目PPT（为路演准备）**
- 游园会得分赛道前列的项目进入路演评审最终奖项
- PPT供路演使用，不作为游园会评审依据

**5. 最终提交通道（必做）**
- 内容包括：项目赛道、项目信息、GitHub项目链接、小红书视频链接等
- 截止时间：最后一天早上8:00
- 提交后无法修改项目信息

## 历史教训

在hackathon活动中，项目提交建议：
- **提交要求提前公布**：在开幕式和选手指南中明确说明，避免最后时刻混乱
- **视频要求灵活**：不要求专业制作，手机录屏也可以，重点是演示功能
- **提交平台简洁**：一个飞书收集表就够了，不需要复杂的提交系统
- **现场提交支持**：安排志愿者在现场帮助选手解决提交问题
- **评分透明**：评分标准在活动开始时就公布，让选手有明确的目标
